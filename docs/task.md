ПРОБЛЕМАТИКА
У об’єднаних територіальних громадах (ОТГ) існує системна
проблема невідповідності між фактичним станом активів
(нерухомість, земля, використання ресурсів) та даними,
зафіксованими в офіційних реєстрах і документах. Ці
розбіжності виникають через відсутність синхронізації
даних, ручні процеси обліку та обмежені можливості
контролю. Це призводить до:
втрати доходів
неефективного управління ресурсами
відсутності прозорості
ЗАВДАННЯ
Потрібно розробити технічне рішення (прототип), яке
автоматизує пошук розбіжностей між юридичними даними
та фактичним станом об’єктів, а також представити
концепцію розвитку продукту, базуючись на життєвому
циклі стартапу, прикріпленому нижче.

1 Опис жи тт є вого ц ик лу с тарта пу (5 0 балів)
1.1 Дослідження та декомпозиція проблеми (10 балів)
Визначення користувача та контексту: наскільки
чітко визначено конкретного користувача (роль у
громаді), процес, в якому виникає проблема, та
обмеження (технічні/юридичні) ( 5 балів )
А наліз стейкхолдерів: хто користується продуктом та
впливає на нього, які їхні потреби ( 3 бали )
Ф окус проблеми: наскільки чітко звужено проблему до
конкретного кейсу без надмірного узагальнення ( 2
бали )
1.2 Розробка MVP та валідація ідей (14 балів)
Обґ рунтування функціоналу M VP
: чому обрані саме ці
функції та як вони вирішують задачу пошуку
розбіжностей ( 5 балів )
Демонстрація працездатності: чи показано, що
рішення працює на наданому датасеті ( 4 бали )
М етрика працездатності: що підтверджує, що продукт
досягає своєї мети (відсоток знайдених
розбіжностей, швидкість аналізу, обсяг оброблених
даних тощо) ( 2 бали )
Покроковий сценарій використання: що саме робить
користувач крок за кроком – від входу до результату
(3 бали )
1.3 Правові межі та відповідність (4 бали)
Відповідність нормативно - правовій базі та межі
повноважень: на яких правових підставах система діє
і чи не перевищує повноважень посадових осіб ( 2
бали )
Дотримання прав власників та користувачів активів:
як система захищає інтереси третіх осіб при
виявленні розбіжностей ( 2 бали )
1.4 План масштабування та бізнес-модель (12 балів)
С тратегія масштабування: як те саме рішення може
працювати для більшої кількості громад; що потрібно
для тиражування ( 4 бали)
Б ізнес - модель та монетизація: як продукт генерує
дохід (підписка, ліцензія, % від ефекту тощо) ( 4
бали )
Оцінка ризиків та бар ' єрів
: технічні, організаційні
або ринкові перешкоди для росту та як їх долати ( 4
бали )
1.5 Довгострокове бачення (10 балів)
Л огіка розвитку продукту
: наскільки реалістично
описано подальший розвиток продукту та його зв’язок
з M VP ( 5 балів )
Додаткові можливості: які нові функції, ринки або
партнерства можуть посилити продукт ( 5 балів )
2 Технічна реалізація (35 балів)
2.1 Робота з даними (10 балів)
І мпорт даних: коректність завантаження Excel/CSV ( 2
бали)
Обробка та нормалізація: обробка пропусків,

ідентифікатори) ( 4 бали)
З іставлення даних: логіка порівняння записів між
реєстрами ( 3 бали)
2.2 Логіка виявлення розбіжностей (10 балів)
К оректність результатів: наскільки правильно
визначені розбіжності ( 3 бали)
Р ізноманітність перевірок: кількість і якість
реалізованих типів розбіжностей ( 3 бали)
Практична цінність: наскільки результати є
корисними для користувача ( 4 бали)
2.3 Повнота виконання завдання: чи задовільняє проєкт

2.4 Зручність використання: зрозумілість інтерфейсу та

(5 балів)
2.5 Демонстрація: чітко показано архітектуру рішення,

3 Презентація (15 балів)

3.2 Залученість команди: всі учасники активно

3.3 Пояснення продукту: наскільки чітко та зрозуміло

бали)
3.4 Відповіді на питання: чіткість відповідей на
