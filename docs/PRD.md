# Назва проєкту: **E-State**
*Гасло: Прозорість у кожному квадратному метрі.*

---

## 1. Концепція продукту
**E-State** — це платформа, що автоматизує аудит активів ОТГ. Вона виступає "містком" між паперовими реєстрами та реальною ситуацією в громаді, залучаючи інспекторів та мешканців до створення єдиного "цифрового двійника" майна.

---

## 2. Детальний User Flow (Шлях даних)

1.  **Звід та аналіз (Back-office):**
    * Землевпорядник завантажує витяги з ДЗК (земля) та ДРРП (нерухомість).
    * Система проводить **Fuzzy Matching** (співставлення за адресою та кадастровим номером).
    * Генерується **"Майстер-таблиця"**, де розбіжності (наприклад, площа в ДРРП > площі в ДЗК) підсвічуються червоним.
2.  **Польова перевірка (Inspector View):**
    * Об'єкти з "червоної зони" автоматично потрапляють у мобільний додаток інспектора.
    * Інспектор виїжджає на місце, робить фото, фіксує фактичний стан (наприклад: "Тут не сад, а діючий ресторан") і вносить дані в систему.
3.  **Оновлення та звітність:**
    * "Майстер-таблиця" оновлюється фактичними даними.
    * Система автоматично розраховує: **Різниця = (Податок з факту) - (Податок за старим реєстром)**.
    * Голова ОТГ та депутати бачать графік зростання бюджету.
4.  **Публічний контроль (Citizen Portal):**
    * Мешканець через ІПН (податковий номер) бачить свій профіль майна.
    * Статус: "Дані синхронізовано" або "Потребує уточнення".

---

## 3. PRD (Product Requirements Document)

### 3.1 Функціональні вимоги (Requirements)
* **Модуль Data Matcher:** Алгоритм порівняння `inner join` за кадастровим номером + `levenshtein distance` для порівняння адрес.
* **Інтерфейс Землевпорядника:** Таблиця з пагінацією, фільтри за типом порушення (цільове призначення, площа, відсутність власника).
* **Модуль Інспектора:** Форма внесення фактичних даних (тип об'єкта, площа, фотофіксація).
* **Модуль Мешканця:** Пошук за РНОКПП (ІПН) з відображенням статусу об'єкта.

### 3.2 Технічний стек (Твій вибір як AI Engineer)
* **Backend:** FastAPI (Python) — для швидкої обробки JSON-даних.
* **Data Core:** Pandas — для маніпуляцій з реєстрами.
* **Frontend:** Streamlit або React (у твоєму улюбленому стилі).
* **База даних:** SQLite або PostgreSQL (для зберігання "Майстер-таблиці" після зведення).

## 4. Бізнес-модель та Масштабування (Розділ 1.4)
* **Монетизація:** SaaS-підписка для ОТГ. Громада платить за "хмарне" зберігання та аналітику.
* **Масштабування:** 1.  Додавання модуля **Супутникового моніторингу** (автоматичне виявлення нових забудов).
    2.  Інтеграція з "Дією" для автоматичного сповіщення мешканців про необхідність оновити дані.
